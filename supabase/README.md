# Supabase setup

## 1. Create the project
- Go to https://supabase.com/dashboard → **New Project**.
- Note the **Project URL**, **anon public** key, and **Project Ref** (`xxxxxxxxxxxx` from the URL).

## 2. Configure environment
Copy `.env.example` → `.env.local` and fill:
```
NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
```

## 3. Install the Supabase CLI (one-time)
```bash
brew install supabase/tap/supabase   # macOS
# or: npm i -g supabase
supabase login
supabase link --project-ref <your-project-ref>
```

## 4. Apply migrations

We use the **Supabase CLI migration system**. All schema lives in
`supabase/migrations/` as timestamped, idempotent SQL files. The CLI tracks
which ones have been applied per-database in `supabase_migrations.schema_migrations`.

### Brand-new database
```bash
npm run db:push
```
Applies `20260419000000_baseline.sql` (the full schema) followed by every
later migration in chronological order.

### Existing database (already provisioned by hand from the old `schema.sql`)
Tell the CLI the baseline is already there, then push only the deltas:
```bash
npm run db:repair-baseline   # marks 20260419000000 as already applied
npm run db:push              # runs only the newer incremental migrations
```

### Day-to-day workflow
```bash
npm run db:new add_some_column          # generates supabase/migrations/<ts>_add_some_column.sql
# edit the new file…
npm run db:push                          # apply to the linked remote project
npm run db:status                        # see what's applied where
```

To diff your local schema against remote and auto-generate a migration:
```bash
npm run db:diff descriptive_name
```

## 5. Enable Google OAuth
- Authentication → Providers → **Google**: enable, paste your OAuth client ID + secret.
- Authentication → URL Configuration → **Site URL**: `http://localhost:3000` (and your prod URL).
- Add redirect URL: `http://localhost:3000/auth/callback`.

## 6. Bootstrap admin
After your first Google sign-in, the trigger inserts a `public.users` row.
If your email is **not** `aman.teferi.80@gmail.com`, promote yourself manually:
```sql
update public.users set role = 'admin' where email = 'you@example.com';
```

## File layout
```
supabase/
├── config.toml                                         # CLI project config
├── schema.sql                                          # legacy snapshot (do NOT edit)
└── migrations/
    ├── 20260419000000_baseline.sql                    # full schema as of 2026-04-19
    └── 20260419120000_incremental_warehouse_loans_clerk_rls.sql
```

- **All future schema changes go in a new migration file** generated by
  `npm run db:new`. Never edit a migration that has already been pushed.
- Migrations are non-destructive by convention — every `CREATE` uses
  `IF NOT EXISTS`, every `ALTER` is wrapped in a guarded `do $$ … end$$` block,
  and every policy is `DROP IF EXISTS` + `CREATE`.

## Notes on the schema
- IDs are `text` (matches the old Firestore-style ID generator).
- `users.id` is `uuid` and FKs to `auth.users(id)`.
- `users.uid` is a generated text alias so old client code reading `user.uid` keeps working.
- All money is `numeric` (consider migrating to integer minor units later).
